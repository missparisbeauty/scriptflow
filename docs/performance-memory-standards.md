# performance-memory-standards.md

---

## 資源使用基本原則

- 不長期保存不必要的大型資料（爬取原始 HTML、完整 API 回應）
- 不重複建立相同的外部服務 client（LLMClient、ImageGenClient、CrawlerClient 各初始化一次）
- 不把本來可局部處理的流程做成整包處理（爬取 → 篩選 → 評分，各階段獨立處理）
- 不允許在多個地方重複快取或重複存放相同資料（今日主題只存一份，不各 service 各自存）

---

## 前端效能與狀態管理

- 爆款候選卡、CTA 變體卡、分鏡列等資料量有限，全量載入；成效追蹤歷史列表超過 50 筆時分段載入
- 分鏡 Tab 切換時不重新請求 API，使用已載入的資料切換顯示
- ✓ 脆 / IG 分鏡切換：切換 CSS class，不重新 fetch
- ✗ 不把後端完整 raw 爬取資料全部傳到前端再篩選
- Tab 切換、分類切換時清除前一個狀態的 loading indicator，不殘留

---

## 後端記憶體與請求處理

- 每次爬取只在記憶體保存「今日候選清單」，不保存完整爬取原始內容
- OpenAI API 呼叫一次只處理一個主題的腳本，不批次累積多主題一起送
- ✓ 09:00 排程：爬取 → 評分 → 篩選 → 寫入 Firestore → 釋放記憶體
- ✗ 不把三平台的 Top 30 全部放在記憶體等待比較，評分完一批就釋放一批
- 合規禁詞掃描在 ScriptService 完成後即釋放，不保存中間結果

---

## 檔案與媒體處理

- 分鏡示意圖由 OpenAI gpt-image-1 產出後直接儲存到雲端儲存，不在後端記憶體保留
- 圖像生成失敗不重試超過 2 次，失敗後前端顯示「示意圖生成中」placeholder
- Word / PDF 匯出在請求時即時產生，不預先批次產生後儲存
- 匯出檔案產生後立即回傳 stream，不暫存到後端磁碟

---

## 外部服務資源使用

- `LLMClient`、`ImageGenClient`、`CrawlerClient` 各在 `infra/` 層初始化一次，service 層不重複建立
- OpenAI API 每次呼叫的 input token 設上限（prompt 樣板 + 三個爆款內容），避免超大輸入
- 爬取失敗重試最多 2 次，超過後記錄失敗平台，不無限重試阻塞排程
- ✓ 09:00 排程爬取：三平台並行呼叫，不依序等待
- ✗ 不在 service 層反覆 `import LLMClient(); LLMClient()` 重建 client

---

## 狀態生命週期

- 今日爆款候選在 09:00 產生，寫入 Firestore，小編登入時從 Firestore 讀取（不保存在記憶體）
- 腳本生成中的中間狀態（OpenAI 串流回應）在完成後即釋放，不保存 context
- 合規掃描結果跟腳本一起存入 Firestore，不單獨維護暫存狀態
- 品牌爆款 DNA 計算完成後寫入 Firestore，計算中間的聚合資料即時釋放

---

## 效能風險預防（高風險模式清單）

以下模式在 ScriptFlow 中明確禁止：

- ✗ 一次把三平台 Top 30（90 篇）全部傳入 OpenAI API 分析（應分批或只傳摘要）
- ✗ 每次小編點擊「生成腳本」都重新爬取（應使用當天已爬取的快取候選）
- ✗ 分鏡示意圖在後端記憶體保存超過單次請求週期
- ✗ 合規禁詞清單每次掃描都重新讀取檔案（應在啟動時載入一次）
- ✗ 成效追蹤頁面一次載入全部歷史紀錄（超過 50 筆時分頁）

---

## 先可放大、再局部優化

- 初版以單一品牌使用為設計，不預先為多品牌做 sharding 或 multi-tenant 設計
- 只要不違反上述高風險禁止模式，初版允許較簡單的實作（如同步爬取再改非同步）
- 效能瓶頸應可局部優化（如只改爬取層改為非同步），不觸動腳本生成流程
